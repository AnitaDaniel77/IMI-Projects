Lesson 19 — Little Lemon Form Breakdown

A form is frequently the only thing standing between a user and their goal. Little Lemon's checkout form has several problems worth unpacking as a case study in what not to do.

Layout problems

Text fields are centre-aligned instead of left-aligned. The checkout button isn't visible. There's no basket and no summary of what the user is buying. The user has no context for what they're confirming.

No progress indicator

Checkout is a multi-step process. Without a progress indicator the user has no idea how far they've come or how much is left. Add one. Let users go back and review previous steps.

Unnecessary fields

The form asks for Middle Initial, Evening Phone, Mobile Phone separately, three address lines plus City, and requires the email address to be typed twice. Luke Wroblewski put it clearly: every question you ask requires the user to parse it, formulate a response and enter it. Remove anything that isn't absolutely necessary.

Required fields and character limits

First Name had no asterisk marking it required, yet submitting triggered the error "First Name must be at least 4 characters." Two problems in one field. First, if a field is required, mark it with an asterisk. Second, a minimum of four characters excludes at least 220 real three-character names. That's not a validation rule, it's a design flaw that tells real people their name is invalid.

The fix: remove the minimum character requirement entirely. Set a maximum of around 50 characters for security and storage reasons, which covers the vast majority of real names without excluding anyone.

Email confirmation

Asking users to type their email twice increases typos, not accuracy. The better approach is to send a confirmation link to the address they entered. Less effort for the user, same result.

CVV with no explanation

CVV and CVC are ambiguous acronyms. Most users don't know what they mean without a prompt. Add a small illustration or tooltip showing it's the three digits on the back of the card.

The common thread

Every problem in this form comes down to the same thing: the form was built around what the system needs, not what the user needs. Unclear errors, jargon, redundant fields, no feedback. If a user can't tell what they did wrong or how to fix it, they leave.
